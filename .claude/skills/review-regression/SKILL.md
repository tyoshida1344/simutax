# Review Regression — デグレチェック

変更が既存機能を壊していないかを確認する。

## 対象の特定

1. 変更ファイルと差分を取得する:

```sh
git diff main --name-only
git diff main
```

2. 変更ファイルの全体を読む
3. 変更ファイルを参照・利用している関連ファイルを特定して読む。特に:
   - 変更したエンジン関数を呼び出しているファイル
   - 変更した型定義を使用しているファイル
   - 変更したパラメータを参照しているファイル

## 確認観点

### エンジン関数の互換性

- エンジン関数（`src/logic/`）のシグネチャ（引数・戻り値）が変わっていないか
- 変更した関数の呼び出し元（`simulate()` や UI のフック）と整合しているか
- 削除・リネームした関数・型が他のファイルで参照されていないか
- 端数処理のルールが税目ごとに正しく維持されているか

### データ互換性

- `src/data/taxParams.2026.json` のキー名・データ構造が `src/data/types.ts` の型定義と一致しているか
- パラメータの追加・変更がエンジン関数で正しく参照されているか
- `SimulatorInput` / `SimulatorResult` のフィールド変更がUIフック・コンポーネントに反映されているか

### 計算ロジックの影響

- 条件分岐の変更がエッジケース（所得0、経費が売上を超える、高所得帯の基礎控除逓減など）の挙動を変えていないか
- 所得税と住民税の控除計算の非対称性（基礎控除、扶養控除額の差異）が維持されているか
- 事業税が青色申告控除を戻した事業所得で計算される仕様が維持されているか
- 社会保険料の自動計算値と手入力上書きの切り替えが正しく動作するか

### UI の影響

- 型定義の変更がコンポーネントの props に反映されているか
- `useSimulator` フックの戻り値の構造が全コンポーネントで正しく参照されているか
- 新しいフィールドの追加時にデフォルト値が設定されているか

### テストの整合性

- 既存テストが変更後も意図通りのケースをカバーしているか
- 新しいロジックに対応するテストが追加されているか
- テストの期待値が変更後の仕様と一致しているか

## 出力

指摘事項を以下の形式で報告する:

```
### デグレチェック結果

- [ ] `ファイル名:行番号` — 指摘内容（影響を受ける箇所: `関連ファイル:行番号`）

指摘なしの場合: 「デグレの懸念は見つかりませんでした」
```

修正が求められている場合（issue-handler 経由）は、報告後に指摘箇所を修正する。
