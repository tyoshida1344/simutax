# Review Perf — パフォーマンスレビュー

変更差分を「コード量を減らせないか」「処理を軽くできないか」の2軸で確認する。

## 対象の特定

1. 変更ファイルを取得する:

```sh
git diff main --name-only
```

2. 変更ファイルの全体を読む（diff だけでなく前後の文脈が必要）
3. 変更ファイルを参照・利用している関連ファイルも特定して読む

## 確認観点

以下の観点で変更箇所とその周辺を確認する。全てを機械的にチェックするのではなく、**実際に効果がある指摘** に絞る。

### コード量の削減

- 既存のユーティリティ関数（`src/logic/common.ts` の `roundDownTo`, `clampMin` 等）で置き換えられる独自実装がないか
- 重複するロジックを関数に抽出できないか
- 不要な変数・関数・条件分岐がないか
- 同じ計算の重複呼び出しを変数にキャッシュできないか
- TypeScript の型ユーティリティ（`Pick`, `Omit`, `Partial` 等）で型定義を簡潔にできないか

### 処理の軽量化

- `useMemo` / `useCallback` が適切に使われているか（不要な再計算の防止）
- 高頻度の入力イベントで重い処理が走っていないか
- コンポーネントの不要な再レンダリングがないか（props の参照安定性）
- ループ内の不変な計算をループ外に出せないか
- エンジン関数内で同じパラメータの走査を複数回行っていないか

## 出力

指摘事項を以下の形式で報告する:

```
### パフォーマンスレビュー結果

**コード量削減**
- [ ] `ファイル名:行番号` — 指摘内容（改善案）

**処理軽量化**
- [ ] `ファイル名:行番号` — 指摘内容（改善案）

指摘なしの場合: 「パフォーマンス上の問題は見つかりませんでした」
```

修正が求められている場合（issue-handler 経由）は、報告後に指摘箇所を修正する。
