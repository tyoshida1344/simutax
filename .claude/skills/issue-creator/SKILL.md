# Issue Creator — イシュー作成スキル

コードベースの調査結果や会話中の課題をGitHubイシューとして登録する。
イシューは「後で着手する作業単位」として機能するため、第三者が読んでも何をすべきか分かる粒度で書く。

## モード判定

ユーザの依頼内容から、以下のどちらのモードで動くか判断する:

- **調査モード**: 「リファクタ余地を調査して」「テスト不足を洗い出して」「改善点を探して」など、
  まずコードを読んで課題を発見する作業が必要な場合
- **登録モード**: 「この課題をイシューにして」「さっきのTODOを登録して」など、
  課題が既に明確で登録だけが必要な場合

判断がつかない場合はユーザに確認する。

## 調査モード

### 1. 調査観点の確認

ユーザが指定した観点（リファクタ、パフォーマンス、テストカバレッジ、設計改善など）を確認する。
観点が曖昧な場合は、具体的にどの範囲・どの層（`src/logic/`、`src/ui/`、`src/data/` など）を
対象とするかをユーザに確認する。

### 2. コードベースの調査

対象ファイルを読み、指定された観点で課題を洗い出す。
調査時は以下を意識する:

- プロジェクトの設計原則（`CLAUDE.md`）に照らして乖離がないか
- 仕様書（`税金シミュレータ仕様書_v1.md`）に記載があるが未実装の項目はないか
- 既存のパターンと一貫性が取れていない箇所はないか

### 3. 調査結果の提示と選別

発見した課題を一覧としてユーザに提示し、どれをイシューとして登録するか確認する。
全てを登録する必要はなく、ユーザが取捨選択できるようにする。

提示形式:
```
1. [課題タイトル] — 概要の1行説明
2. [課題タイトル] — 概要の1行説明
...
```

ユーザが選択したもの、または「全部登録して」と言われた場合は全てを登録に進む。

### 4. イシュー登録

選択された課題をイシューとして登録する（後述の「イシューの書き方」に従う）。

## 登録モード

### 1. 課題の整理

会話の文脈から課題の内容を整理する。以下が不明確な場合はユーザに確認する:

- 何を変更・追加・修正するのか
- 対象のファイル・機能・モジュールはどこか

### 2. イシュー登録

整理した内容をイシューとして登録する（後述の「イシューの書き方」に従う）。

## イシューの書き方

`.github/ISSUE_TEMPLATE/default.md` を読み、そのテンプレートに沿って本文を作成する。
テンプレートが変更されている可能性があるため、毎回必ずファイルを読んでから書き始める。

### ルール

- **タイトル**: 動詞で始める簡潔な1行（例: 「消費税3方式の比較機能を追加」「基礎控除の逓減ロジックを検証」）
- **概要**: 背景と目的を第三者にも伝わるように書く。会話の文脈に依存した「さっきの件」のような表現は避ける
- **対応内容**: 着手する人が作業を開始できる粒度。ファイル名やモジュール名が分かっていれば含める
- **参照**: 仕様書のセクション番号（例: `仕様書 4.3`）、関連ファイルパス、関連イシュー番号（`#1` など）があれば記載。なければ省略可

### 粒度の判断

1つのイシューが複数の独立した作業を含む場合は分割する。
逆に、密接に関連する小さな変更は1つにまとめる。
判断に迷う場合はユーザに確認する。

## 登録コマンド

テンプレートの各セクションを埋めた内容を `--body` に渡す。

```sh
gh issue create --title "<タイトル>" --body "$(cat <<'EOF'
<テンプレートの各セクションを埋めた内容>
EOF
)"
```

複数イシューを登録する場合は、登録前にタイトル一覧をユーザに最終確認してから実行する。

## 完了報告

登録したイシューのタイトルとURLを一覧で報告する:

```
登録したイシュー:
- #4 消費税3方式の比較機能を追加 — <URL>
- #5 基礎控除の逓減ロジックを検証 — <URL>
```
