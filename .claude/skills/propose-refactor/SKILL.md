# Propose Refactor — リファクタリング提案

プロジェクト全体を「アーキテクチャ」「コード品質」「パフォーマンス」「保守性」「依存関係・技術選定」の5軸で分析し、優先度付きのリファクタリング提案を行う。

review-perf / review-regression / review-ux が「直近の変更差分」をレビューするのに対し、このスキルは **コードベース全体** を対象にプロジェクトの健全性を評価する。変更差分に依存しないため、開発の節目や方針転換時に使う。

## 1. プロジェクト規約の把握

分析の前に「このプロジェクトが何を目指しているか」を理解する。こちらの主観ではなく、プロジェクトが自らに課したルールとの乖離を測定するため。

1. `CLAUDE.md` を読み、設計原則・ディレクトリ構成・命名規約・技術スタックを把握する
2. `.claude/rules/` 配下のルールファイルを読む
3. 仕様書（`税金シミュレータ仕様書_v1.md` 等）がある場合、UIデザイン指針と実装順序を確認する

## 2. コードベースの全体把握

ファイル構成とモジュール間の依存関係を把握する。全ファイルを読む必要はないが、各ディレクトリの代表的なファイルには目を通す。

1. ディレクトリ構成を確認する（`CLAUDE.md` が想定する構成と実際の構成の対応を意識する）
2. エントリポイント（`main.tsx`, `App.tsx`）からデータの流れを追う
3. エンジン層（`src/logic/`）の関数シグネチャと依存関係を確認する
4. UI 層（`src/ui/`）のコンポーネント構成と状態管理パターンを確認する
5. データ層（`src/data/`）の型定義とパラメータ構造を確認する
6. テストファイルのカバレッジと分布を確認する

## 3. 5軸分析

各軸で以下の観点を確認する。機械的に全項目を列挙するのではなく、**実際に効果がある指摘** に絞る。「直せば明確に良くなる」ものだけを挙げる。

### 3-1. アーキテクチャ

プロジェクトの構造的な健全性を評価する。

- **ディレクトリ構成の一貫性**: `CLAUDE.md` が定めた構成と実際の構成が一致しているか。ドキュメントと実装のドリフト（命名の食い違い等）がないか
- **モジュール境界**: エンジン層が UI に依存していないか。逆に UI がエンジンの内部実装に依存していないか
- **データフロー**: 入力 → 計算 → 表示の流れが一方向になっているか。双方向の依存や循環参照がないか
- **責務の分離**: 1つのファイル/関数が複数の責務を持っていないか（例: オーケストレータが個別の計算ロジックも含む）
- **パラメータ注入の徹底**: 税率・控除額・閾値がすべてパラメータ JSON に外出しされているか。ハードコードされた定数がないか

### 3-2. コード品質

読みやすさ・型安全性・一貫性を評価する。

- **DRY 違反**: 同じロジックが複数箇所に書かれていないか。同じバリデーション/変換/制約が UI とエンジンの両方で行われていないか
- **型安全性**: `as unknown as T` のような危険なキャスト、`any` の使用、未検証の外部データ読み込みがないか
- **ミュータビリティ**: 関数の戻り値を呼び出し元が書き換えるなど、純関数パターンを崩している箇所がないか
- **命名の一貫性**: 変数名・関数名・ファイル名の命名規則がプロジェクト内で統一されているか
- **コーディングルール準拠**: `.claude/rules/` に定められたルール（CSS の余白値、margin 方向等）に違反がないか
- **死んだコード**: 使われていない関数・変数・型定義・import がないか

### 3-3. パフォーマンス

実行時の効率とバンドルサイズを評価する。

- **不要な再計算**: 入力変更時に無関係な計算まで走っていないか。`useMemo` / `useCallback` の適用が適切か
- **不要な再レンダリング**: props の参照が不安定で子コンポーネントが毎回レンダリングされていないか
- **計算の重複**: エンジン内で同じパラメータ走査・ブラケット検索を複数回行っていないか
- **バンドルサイズ**: 不要な依存パッケージ、tree-shake できない import がないか

### 3-4. 保守性

将来の変更・拡張のしやすさを評価する。

- **テストカバレッジ**: エンジンの各関数にテストがあるか。境界値・エッジケース（所得0、経費＞売上等）がカバーされているか。UI 層のテスト方針は適切か
- **拡張容易性**: 新しい税目・控除項目・計算方式を追加するとき、変更が何ファイルに波及するか。パラメータ JSON に項目を追加するだけで済む設計か、コードの変更も必要か
- **設定のハードコード**: 将来変わりうる値（年度、適用期限、閾値）がコード中に埋め込まれていないか
- **ドキュメントとコードの整合**: `CLAUDE.md` や仕様書の記述が実装と乖離していないか

### 3-5. 依存関係・技術選定

現在の依存パッケージの妥当性と、ライブラリの導入・置換による改善余地を評価する。

このプロジェクトは「依存は最小限」を方針としているため、ライブラリ導入提案には必ず **導入コスト vs 得られる効果** のトレードオフ評価を付ける。「便利だから入れる」ではなく「手書きで維持するコスト/リスクがライブラリ導入コストを上回る」場合にのみ提案する。

- **型安全性の強化**: 外部データ（JSON, API レスポンス等）の読み込みに型保証がない箇所で、バリデーションライブラリ（Zod 等）が有効か。バンドルサイズと導入コストとのトレードオフ
- **テストユーティリティ**: テストの記述量やデータ準備を削減するライブラリ（testing-library 等）が有効か。現在のテスト戦略（エンジン純関数のユニットテストのみ）との整合性
- **ビルド・開発ツール**: リンター・フォーマッター・型チェックのツールチェーンに改善余地がないか。既存ツール（OxLint 等）で十分かどうか
- **UI ユーティリティ**: フォーム管理・状態管理・アニメーション等で、手書きコードの量やバグリスクを減らせるライブラリがないか。ただし SPA の規模に対して過剰でないか
- **既存依存の見直し**: 現在の依存パッケージにメンテナンスが止まっているもの、より軽量な代替があるもの、不要になったものがないか

トレードオフ評価の項目:
- **バンドルサイズへの影響**: gzip 後のサイズ増加（tree-shake 可能かどうかも考慮）
- **学習コスト**: プロジェクトの既存パターンとの親和性
- **メンテナンスリスク**: ライブラリの更新頻度・コミュニティ規模・破壊的変更の頻度
- **代替手段**: ライブラリなしで同等の効果を得る方法があるか（あるなら比較）

## 4. 出力

分析結果を以下の形式でレポートする。

```
## リファクタリング提案レポート

### サマリ

<プロジェクト全体の健全性を2〜3文で要約。良い点も明記する>

### 提案一覧

#### [High] カテゴリ名 — 提案タイトル
- **現状**: `ファイル名:行番号` — 現在どうなっているか
- **問題**: なぜ変えるべきか（規約違反 / バグリスク / 保守コスト増 等）
- **提案**: 具体的にどう変えるか
- **影響範囲**: 変更が波及するファイル / モジュール
- **工数感**: 小（〜30分） / 中（1〜2時間） / 大（半日以上）

#### [Medium] ...

#### [Low] ...

### 対象外としたもの

<意図的に指摘しなかった項目があれば理由とともに記載。例: 「将来の M4 実装まで判断を保留」等>
```

優先度の基準:
- **High**: 規約違反、バグリスク、データの不整合など放置すると問題が拡大するもの
- **Medium**: コードの可読性や保守性を明確に改善するもの
- **Low**: あると良いが、現状でも実害はないもの

同じ優先度内では、工数が小さく効果が大きいものを先に置く。

## 5. イシュー登録

レポート出力後、提案一覧の各項目をイシューとして登録するか確認する。

1. 提案一覧を番号付きリストで提示し、どれをイシュー化するかユーザに選択してもらう（「全部」「High だけ」「1, 3, 5」等）
2. ユーザが選択したら `/issue-creator` を登録モードで呼び出し、選択された提案をイシューとして登録する
3. イシューのタイトルは提案タイトルをそのまま使い、本文にはレポートの該当セクション（現状・問題・提案・影響範囲・工数感）を転記する

ユーザが「登録しない」と言った場合はレポートのみで終了する。

## やらないこと

- ファイルの編集・コミット・PR 作成（本スキルはレポートのみ。実装は issue-handler の役割）
- 変更差分のレビュー（それは review-perf / review-regression / review-ux の役割）
- 仕様そのものへの提案（「この税目の計算方法を変えるべき」等は範囲外）
- 技術スタックのフレームワーク級の変更提案（「React をやめて Vue にしよう」等は範囲外。スタック内でのライブラリ追加・置換・削除は 3-5 軸で評価する）
